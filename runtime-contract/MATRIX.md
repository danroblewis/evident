# The dual-engine behavior-contract matrix

> Generated by `runtime-contract/run-matrix.sh` (re-run it to regenerate). This
> is the keystone proof of the hybrid-integration session: **both runtime-split
> strategies reproduce the captured per-tick semantics** of the current Evident
> multi-FSM runtime, fixture by fixture, with every boundary documented (never
> faked).

## What's being proved

`runtime-contract/fixtures/` captures 15 single-tick behaviors of the current
runtime as implementation-agnostic SMT-LIB + metadata + golden (next-state model
+ dispatched effects). An engine "reproduces" a fixture when it computes the same
next-state (and, where it surfaces them, the same effects). The harness crate
`runtime-contract` (lib) defines one `FsmEngine` trait + a matrix runner; every
engine plugs in by implementing `tick(&Fixture) -> Outcome` and going green.

**Verdict legend** — the only red verdict is `✗` (a *wrong* answer):

| glyph | meaning |
|---|---|
| `✓`  | full match: next-state model **and** effects reproduced |
| `✓ˢ` | state verified; effects not encoded in the portable SMT for this fixture (they're `effects_in_smt:false`, checked against the runtime engine instead) |
| `—`  | **gap**: engine cleanly declined a fixture it can't do yet (a *documented* capability boundary — green, not a failure) |
| `✗`  | **fail**: engine produced a wrong answer (the only failing verdict; zero across all engines) |

## Reading the result

- **Strategy 1 (greenfield) reproduces all 15** — full state+effects on the 8
  `effects_in_smt:true` fixtures, state-only (`✓ˢ`) on the 5 positives whose
  effects the portable capture leaves to the runtime engine, and the genuine
  UNSAT witness on the 2 negatives. Zero wrong answers.

- **Strategy 2 has two columns by design — the progression is the finding.**
  - **v1 (scalar):** reproduces only the scalar-state slice (`prev_record_fields`)
    + the negatives. The 12 `—` are all the *same* documented boundary:
    enum-typed `state` driven by an SMT-LIB `(declare-datatypes …)`. The v1
    scalar path can't read that output (the duplicate-`DatatypeSort`
    entanglement), so it cleanly declines — an honest gap, not a fake pass.
  - **enum-increment:** the boundary is crossed with one additive function
    (`smtlib_fsm::solve_smtlib_decode_all`) that decodes the solved model's
    constants generically via raw `z3-sys` — scalars, datatypes (→ enum), and
    sequences (→ effect list) — needing no registered sort. With it, **strategy
    2 matches strategy 1 exactly (10 ✓ / 5 ✓ˢ).**

- **Both strategies, by two independent code paths, land on the same 10 ✓ / 5 ✓ˢ
  / 0 ✗** — and that is, in turn, byte-consistent with the CurrentRuntime gate
  and the pure-Z3 SMT-LIB faithfulness proof. The captured semantics are
  reproduced under both "rewrite the engine" and "evolve the engine."

## Convergence — the front-to-back pipeline

Beyond the per-tick matrix, the full hybrid pipeline runs end-to-end:

```
Evident FSM source → [runtime_smt::transpile_fsm] → SMT-LIB + metadata
                   → [greenfield engine: scheduler::run] → stdout + exit code
```

`runtime-smt/tests/convergence.rs` + `runtime-smt/crosscheck.sh` prove this is
**byte-identical** to the legacy binary (`evident effect-run`) on:

| Evident source | stdout | exit |
|---|---|---|
| `runtime-smt/crosscheck/countdown.ev` (scalar, multi-tick) | `tick`/`tick`/`tick`/`done` | 0 |
| `examples/test_08_exit_code.ev` (**real example**, enum state) | `exiting with code 42` | 42 |
| `examples/test_03_seq_chain.ev` (**real example**, enum state) | `first`/`second`/`third` | 0 |

So the greenfield engine + a real Evident→SMT-LIB front-end *is* a working
"next runtime" for the subset it covers.

## Honest boundaries (not faked)

- `✓ˢ` is not `✓`: for the 5 `effects_in_smt:false` positives, the portable
  SMT capture deliberately doesn't encode effects (they're string/`last_results`
  -dependent and were checked against the runtime engine when captured). Both
  SMT-ingesting strategies therefore verify *state* on those and leave effects
  to the runtime-engine column — exactly what the contract's own TODO #6 records.
- The greenfield engine and strategy-2's enum-increment cover the per-tick
  transition relation. Async event sources (FrameTimer/Stdin/Sigint),
  `last_results` *threading* across ticks, FFI effects, and mode-2 toposort
  dispatch remain out of scope for the SMT-LIB engines (see
  `runtime-smt/README.md` TODO and `runtime-contract/README.md` contract-TODOs).
  No fixture covering those was forced into a fake pass.
