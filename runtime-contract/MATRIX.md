## What's being proved

`runtime-contract/fixtures/` captures 22 single-tick behaviors of the current
runtime as implementation-agnostic SMT-LIB + metadata + golden (next-state model
+ dispatched effects). An engine "reproduces" a fixture when it computes the same
next-state (and, where it surfaces them, the same effects). The harness crate
`runtime-contract` (lib) defines one `FsmEngine` trait + a matrix runner; every
engine plugs in by implementing `tick(&Fixture) -> Outcome` and going green.

**Verdict legend** — the only red verdict is `✗` (a *wrong* answer):

| glyph | meaning |
|---|---|
| `✓`  | full match: next-state model **and** effects reproduced |
| `✓ˢ` | state verified; effects not encoded in the portable SMT for this fixture (checked against the runtime engine instead) |
| `—`  | **gap**: engine cleanly declined a fixture it can't do yet (a *documented* capability boundary — green, not a failure) |
| `✗`  | **fail**: engine produced a wrong answer (the only failing verdict; zero across all engines) |

## Reading the result

- **Both runtime-split strategies — greenfield (S1) and enum-increment (S2) —
  reproduce all 22 fixtures fully: 22 ✓ / 0 ✓ˢ / 0 ✗.** Every captured behavior,
  including the four formerly-open boundaries, is reproduced by both independent
  code paths, and is byte-consistent with the CurrentRuntime gate and the pure-Z3
  faithfulness proof.

- **The v1-scalar column (engine 4) is the deliberate baseline.** It reads only
  scalar outputs (`Int`/`Bool`/`Real`/`Str`); any fixture with enum-typed `state`
  driven by an SMT-LIB `(declare-datatypes …)` is its documented entanglement
  boundary, so it cleanly declines (`—`) — an honest gap, never a fake pass. It
  resolves `prev_record_fields` (scalar record fields, no enum state) and the two
  negatives. Engine 5 (enum-increment) crosses that boundary with one additive
  generic decoder (`solve_smtlib_decode_all`) and reaches full parity.

## The four boundaries — now closed

The hybrid-integration session left four behaviors out of the SMT engines'
reach. Each is now captured as a deterministic single-tick fixture, derived from
the real Evident path and reproduced by both SMT engines:

| Gap | Fixture(s) | What is captured |
|---|---|---|
| **`last_results` threading** | `feedback_format_tick`, `feedback_parse_read`, `feedback_int_payload` | The effects Seq driven by a `last_results` read is now **encoded in the portable SMT** (was `effects_in_smt:false` → `✓ˢ`); `feedback_int_payload` adds Int-**payload** threading (`IntResult(41)` → `IntToStr(42)`). Greenfield also threads `last_results` across ticks end-to-end (see `runtime-smt/fixtures/feedback_loop.smt2`). |
| **FFI effects** | `ffi_libcall_emit`, `ffi_shellrun_exit` | Effect EMISSION beyond `Println`/`Exit`: `LibCall("libc","abs","i(i)",⟨ArgInt(-7)⟩)` (Effect ADT carries a `(Seq FFIArg)` payload, decoded generically) and `⟨ShellRun("echo hi"), Exit(0)⟩`. The per-tick unit is the emitted effect list; the C side effect is dispatch, not transition. |
| **async event sources (read side)** | `async_signal_exit`, `async_tick_count_read`, `async_stdin_line_read` | How a user FSM **reads** each plugin-as-writer field (`signal_received` / `tick_count` / `stdin_line`) as a deterministic transition. |
| **mode-2 dispatch** | `mode2_ordered_chain` | An FSM with **no `effects` slot** whose Effect bindings are **toposorted** by a `Seq(Effect)` ordering declaration. The CurrentRuntime gate witnesses the *real* toposort via `EvidentRuntime::collect_tick_effects` (`dispatch_mode:2`); the SMT capture encodes the resulting order. |

## Honest boundaries (still out of scope, not faked)

These remain genuinely outside a deterministic single-tick fixture. None were
forced into a fake pass; they are documented as bigger sub-projects.

- **The async *injection* (the awaiter).** The fixtures above capture how an FSM
  *reads* an injected world field; producing that field from wall-clock / signal /
  blocking `read(2)` is the generic-awaiter sub-project
  (`docs/design/event-sources-as-evident.md`). It depends on real time / external
  input and has no satisfying assignment to compute — outside a transition fixture.
- **mode-2 random tie-break for *partial* orders.** `mode2_ordered_chain` uses a
  **total** order (edges `a→b→c`), which is seed-independent. A *partial* order
  leaves the runtime's `EVIDENT_DISPATCH_SEED` random tie-break free; capturing it
  needs a pinned seed and a fixture format that records the edge graph.
- **Cross-tick / multi-tick properties.** Cross-FSM wake propagation, same-tick
  writer→reader visibility, and `run(F, init)` percolated child effects are
  multi-tick properties of `run_with_ctx`, not a single `query_with_pins_and_given`.
  A future multi-tick fixture format would capture them.
- **`Time`/`MonotonicTime` in the greenfield threading loop** are a deterministic
  stub (the engine has no wall clock; a real one would break byte-identical
  cross-checks). Documented in `runtime-smt/src/effect.rs`.

## Convergence — the front-to-back pipeline

Beyond the per-tick matrix, the full hybrid pipeline runs end-to-end:

```
Evident FSM source → [runtime_smt::transpile_fsm] → SMT-LIB + metadata
                   → [greenfield engine: scheduler::run] → stdout + exit code
```

`runtime-smt/tests/convergence.rs` + `runtime-smt/crosscheck.sh` prove this is
**byte-identical** to the legacy binary (`evident effect-run`) on:

| Evident source | stdout | exit |
|---|---|---|
| `runtime-smt/crosscheck/countdown.ev` (scalar, multi-tick) | `tick`/`tick`/`tick`/`done` | 0 |
| `examples/test_08_exit_code.ev` (**real example**, enum state) | `exiting with code 42` | 42 |
| `examples/test_03_seq_chain.ev` (**real example**, enum state) | `first`/`second`/`third` | 0 |
