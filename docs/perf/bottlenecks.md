## How to read this report

Every queryable schema in `examples/` and `stdlib/` is profiled with [`evident profile`](../../runtime/src/commands/profile.rs). For each claim the tool:

1. **Solves it once, cold** (translate → build solver → Z3 `check()` → extract a model) and times the whole thing. That wall-clock is the claim's **baseline solve cost** — column *baseline solve (μs)* below. The sweep runs with `EVIDENT_FUNCTIONIZE=0`, so this is the **pure Z3 cost**: exactly the work the JIT / native-codegen path aims to replace.

2. **Ranks the solved-for variables by pin-savings.** It pins each scalar (`Int`/`Bool`/`String`) leaf the baseline model produced to that value and re-solves. *savings(μs)* = `baseline − pinned`: how much faster the solve gets when a caller (or an upstream FSM) supplies that variable instead of leaving Z3 to derive it. A large positive savings means **that variable is a bottleneck** — feeding it in, caching it, or compiling its derivation would pay off most. `Δconflicts` / `Δdecisions` are the matching drops in Z3 search effort (`baseline − pinned`); they corroborate *why* the pin helped.

**Caveats.** Times are single-run, warm-process microsecond solves on one machine — treat them as *relative* signal, not absolute benchmarks. Small claims sit near solver-overhead noise (±a few hundred μs); a negative savings just means the pin cost more than it saved at that scale. Claims marked **no scalar candidates** solve for only enum / sequence / record values (e.g. an FSM whose outputs are `state_next` + `Seq(Effect)`); the v1 pin-finder only pins scalars, so there's no per-variable table — the baseline solve cost is still reported. `unsat` / `error` claims couldn't produce a satisfiable baseline standalone (often they need scheduler-level pins) and are listed for completeness.
