# Automata Testing Patterns — Research Notes for Evident

## The Landscape

The literature on testing finite state machines spans three largely separate
traditions: *conformance testing* (does the implementation match the
specification?), *model checking* (does any execution violate a temporal
property?), and *property-based testing* (does the system behave correctly on
randomly generated inputs?). All three are relevant to Evident's step executor,
and each maps differently onto what Evident already does.

---

## 1. FSM Conformance Testing: W-Method and Descendants

The W-method (Chow 1978; independently Vasilevskii 1973) answers: "what minimal
set of input sequences proves that an implementation has the same states and
transitions as the specification?" Its guarantees are strong but its prerequisites
are stringent.

**How the W-method works.** Given a reduced, deterministic, completely specified
specification FSM with `n` states:

1. Construct the *characterization set* W — a set of input sequences such that
   every pair of distinct states produces different outputs on at least one
   sequence in W. (These are *distinguishing sequences* — they identify which
   state you are in by observing outputs.)
2. For each state `s` and each transition `(s, a, s')`, the test suite must
   include a sequence that drives the machine to `s`, fires `a`, and then
   applies every sequence in W to confirm the post-transition state is `s'`.
3. If the implementation has at most `n + k` states (for some extra-states
   budget k), the resulting test suite detects all faults.

The Wp-method (Fujiwara et al. 1991) reduces suite length by using *state
identifiers* only for the target state rather than applying the full W-set
everywhere. The HSI-method and H-method tighten this further. All share the
same fundamental guarantee: the test suite is *complete* with respect to a
bounded fault model, meaning every incorrect implementation is caught.

**What this requires that Evident does not yet have.** Conformance testing
assumes (a) a reduced, deterministic specification, (b) the ability to reset the
implementation to a known initial state, and (c) deterministic, observable
outputs on every transition. Evident schemas are nondeterministic by nature — the
solver may return any satisfying assignment — and the step executor does not yet
expose a reset mechanism. More critically, W-method test suites are generated by
offline analysis of the specification graph, which requires enumerating all
states: feasible for small enum-state machines, but Evident schemas can have
infinite or symbolically large state spaces.

**What is actionable.** For Evident automata with finite, fully enumerated state
types (e.g., `type GameState = Lobby | Playing | GameOver`), a W-method suite
could be constructed by hand or by a future tool. The `unsat_` claim mechanism
already supports the key W-method assertion: "after this sequence of transitions,
state X is impossible." A multi-step `unsat_` claim is exactly a distinguishing
sequence check. The missing piece is tooling to generate these sequences
automatically from the schema definition.

## 2. Coverage Criteria for FSMs

Coverage criteria impose a partial order on test suite quality:

- **State coverage**: every state appears in at least one execution. Weakest
  useful criterion. Achievable by `sat_` claims that exercise each state.
- **Transition coverage**: every transition (state × input → next_state) fires
  at least once. The practical minimum for serious FSM testing. A `sat_` claim
  per transition achieves this.
- **Transition-pair coverage** (also called N+1 switch cover): every pair of
  consecutive transitions fires. Catches faults where a state is entered
  correctly but leaves incorrectly depending on how it was entered.
- **All-paths coverage**: every path through the FSM fires at least once.
  Infeasible in general (cycles make the path count infinite); bounded to paths
  of length ≤ k.

Empirical studies show that transition coverage subsumes state coverage, and
transition-pair coverage catches significantly more bugs than transition coverage
alone. All-paths quickly becomes intractable: for FSMs with more than nine
states, the percentage of test suites that are n-complete (catch all n-extra-state
faults) drops sharply.

**In Evident today.** Transition coverage can be expressed with one `sat_` claim
per transition. Transition-pair coverage requires two chained transition variables
in the claim body — `t1 ∈ Transition`, `t2 ∈ Transition`, `t2.state = t1.next`,
specific state/input bindings on both. This works now. What is missing is
*measurement*: a tool that computes which transitions a test suite covers, and
flags uncovered ones.

## 3. Model-Based Testing and the Oracle Problem

Utting and Legeard's framework (Practical Model-Based Testing, 2007) names the
fundamental difficulty: *the oracle problem*. A test case is only a test if you
know what the correct output should be. For stateful systems, this requires
either (a) a reference implementation, (b) a formal specification that serves as
the oracle, or (c) a property that is checkable without knowing the exact output.

QuickCheck-state-machine (Haskell) and Hypothesis's stateful testing (Python)
both solve the oracle problem by maintaining a *model* — a simplified pure
implementation that the tester writes alongside the real system. Generated command
sequences are replayed on both; divergence is a failure. The shrinker reduces
failing sequences to minimal counterexamples by removing commands while
respecting dependency ordering (file must be opened before it is read).

**Evident's position.** Evident sidesteps the oracle problem in an important way:
the schema *is* the specification, and the solver confirms that any assignment
it returns satisfies all constraints. The oracle is built in. `sat_` and `unsat_`
claims do not need a reference implementation because the constraint system itself
arbitrates correctness. This is a genuine advantage over imperative stateful
testing — you cannot accidentally test against a wrong oracle.

What Evident lacks is the *trace generation* half: automatically producing
diverse multi-step sequences and checking that each step is satisfiable.
`evident sample` does this for single steps. The step runner under design
extends this to full traces. Property-based stateful testing (QuickCheck/
Hypothesis style) then becomes: generate a random sequence of inputs, run each
through the step executor, and check that `unsat_` invariants hold after every
step. Shrinking is hard — Evident cannot yet programmatically search for minimal
failing prefixes without external tooling.

## 4. Model Checking and Temporal Logic

TLA+, SPIN, and NuSMV take a different approach: rather than generating test
suites, they *exhaustively* verify that every reachable state satisfies a
temporal property. The properties are written in LTL (Linear Temporal Logic,
path quantification) or CTL (Computation Tree Logic, tree-branching quantification):

- LTL: `□(state ≠ Error)` — *always* not in error state (safety)
- LTL: `□(requested → ◇ granted)` — *always* if requested then *eventually*
  granted (liveness)
- CTL: `EF(state = Done)` — *there exists* a path that *eventually* reaches Done

Model checkers enumerate or symbolically represent the full reachable state space.
For FSMs with more than ~10^8 states, they use *symbolic model checking* with
BDDs or SMT solvers rather than explicit enumeration. When a property fails, the
tool produces a counterexample trace.

**Bounded model checking (BMC)** is the subset most relevant to Evident. BMC
unrolls the transition relation `k` times and encodes the question "does any
execution of length ≤ k violate property P?" as a single SMT formula:

```
INIT(s₀)
∧ T(s₀, s₁)
∧ T(s₁, s₂)
∧ ...
∧ T(s_{k-1}, sₖ)
∧ ¬P(s₀) ∨ ¬P(s₁) ∨ ... ∨ ¬P(sₖ)
```

If this formula is satisfiable, the satisfying assignment is a counterexample.
If it is unsatisfiable for all k up to the *completeness threshold* (the diameter
of the reachable state space), the property is proved. Tools like ESBMC and
NuSMV/nuXmv implement this with Z3 or similar SMT backends.

**This is exactly what Evident already does for finite traces.** An Evident
`unsat_` claim with `t1 ∈ Transition`, `t2 ∈ Transition`, ..., `tₖ ∈ Transition`
chained together is BMC at depth k. The chain of transition variables, linked
by `t_{i+1}.state = t_i.next`, *is* the unrolled transition relation. The
`unsat_` prefix asserts that no satisfying assignment exists — which is the BMC
safety check. Evident is not reinventing BMC; it is independently discovering it
through the constraint-schema idiom.

The gap between current Evident and full BMC tooling: (a) generating these
chained claims automatically up to depth k, (b) increasing k until the
completeness threshold is reached or a counterexample is found, and (c) returning
the counterexample trace when UNSAT flips to SAT. None of this requires new
language features — only a runner that constructs and submits the chained queries.

## 5. What Evident Can Already Do vs. What Requires New Work

### Already achievable with `sat_`/`unsat_` claims

- **Single-step transition testing**: `sat_` confirms a transition is reachable;
  `unsat_` confirms it is impossible.
- **State-pair distinguishing**: write two transitions from the same state on the
  same input and assert their next-states differ (or don't) — this is W-method
  reasoning by hand.
- **Bounded reachability (BMC at fixed k)**: chain k transition variables,
  assert a property on `tₖ.next` — if `unsat_`, the state is unreachable in k
  steps.
- **Invariant checking at fixed depth**: add `tᵢ.next.state ≠ ErrorState` for
  each i — the `unsat_` prefix checks the invariant holds across the whole chain.
- **Deadlock detection**: assert that no valid transition exists from a given
  state — an `unsat_` claim that tries and fails to find any next-state.

### Requires the step runner (under design)

- **Trace execution and observation**: running the machine on a concrete input
  sequence and recording (state, input, output) at each step.
- **I/O-driven conformance testing**: feeding a W-method test sequence in and
  comparing actual vs. expected outputs step by step.
- **Liveness checking at runtime**: detecting that "good state is never reached"
  during a long run (not expressible as a bounded claim).

### Requires features not yet in Evident

- **Automatic distinguishing sequence generation**: analyzing the schema to
  derive W-sets requires graph algorithms over the state space — needs either
  a dedicated tool or a Datalog fixpoint computation in Evident.
- **Unbounded BMC / k-induction**: proving safety for *all* k (not just fixed k)
  requires iterating k upward and checking k-induction; this is a runner-level
  loop, not expressible in a single claim.
- **Shrinking of failing traces**: finding the shortest prefix that demonstrates
  a violation requires binary search over the trace length — this is a runner
  feature analogous to QuickCheck's shrinker.
- **LTL/CTL property syntax**: temporal operators (`□`, `◇`, `U`) are not
  current Evident syntax. Safety (`□P`) is expressible as `unsat_` chains;
  liveness (`◇P`) requires the step runner to observe eventual outcomes.

## 6. Where Evident Is Novel

The mainstream testing tools either (a) operate on concrete implementations with
an abstract model as oracle, or (b) verify abstract specifications with no
connection to running code. Evident collapses this distinction: the schema is
both the specification *and* the executable program. Testing an Evident automaton
does not compare implementation against model — it confirms that the model's
constraint system is shaped correctly, and execution is just one more query.

This is closest in spirit to TLA+: both describe transitions as predicates over
primed/unprimed variables, and both use a solver to check properties. The
difference is that TLA+ is a pure specification language (you verify it
separately from running it), while Evident schemas are run directly by the step
executor. Evident is, in effect, a *runnable TLA+* where Z3 serves as both
verifier and interpreter.

The risk in this novelty: conformance testing methods assume separation between
the specification and implementation. When they are identical, "conformance
testing" reduces to "does the solver work correctly?" — which is Z3's
responsibility, not the programmer's. The programmer's testing burden shifts
entirely to verifying that the schema correctly captures the intended semantics.
`sat_`/`unsat_` claims are the right tool for this. The W-method's insight —
that distinguishing sequences reveal semantic errors — applies directly: write
`unsat_` claims that assert two states cannot be confused by any sequence of
inputs, and you have a constraint-native W-method.
