# Composition Notations: A Survey for Language Design

**Context:** We are designing a programming language where the central operation is composing constraint systems together. We need a symbol that means "join these constraint systems," ideally: symmetric (not directional), reads left-to-right, visually lightweight, has horizontal spread.

---

## 1. Standard Category Theory Composition

### The Ring Operator ∘ (U+2218)

The classic composition operator. `f ∘ g` means "first apply g, then apply f." This is **applicative order** (right-to-left), mirroring mathematical function application where the innermost is evaluated first.

```
f ∘ g ∘ h    -- means h first, then g, then f
```

The reversal is deliberate: it mirrors `f(g(h(x)))` in expression form. This is firmly entrenched in most of mathematics.

**Directionality:** Right-to-left. The text sequence opposes the execution sequence. Confusing for constraint systems where you want to name components left-to-right.

### The Semicolon ; (U+003B) — Diagrammatic Order

The semicolon notation `f ; g` means "first apply f, then apply g." This is **diagrammatic order** (left-to-right), matching the way arrows flow in categorical diagrams. It originates in Ernst Schröder's relational algebra (1895) and was popularized in category theory and theoretical computer science.

Standard equivalence: `f ; g ≡ g ∘ f`

```
f ; g ; h    -- means f first, then g, then h
```

**Directionality:** Left-to-right. Reads in execution order. Used heavily in proof assistants (Coq's tactic composition, Lean) and in relational mathematics (Gunther Schmidt's formalization).

### The Middle Dot · (U+00B7 or U+22C5)

Used in some algebraic presentations as a monoid operation — composition without indicating direction explicitly. `f · g` often means the same as `f ∘ g` in context, but in a monoid this is just "multiply."

```
A · B · C
```

**Visual note:** Lighter than ∘ but visually very small — essentially invisible at small sizes.

## 2. String Diagrams — Graphical Category Theory

String diagrams are the visual language of monoidal category theory. A morphism `f : X → Y` is drawn as a box labeled `f`, with wires (strings) for inputs entering from below and outputs exiting at the top (or top-to-bottom depending on convention).

**Composition rules:**
- **Sequential composition** = **vertical stacking**: the output wire of f plugs into the input wire of g. Represents `g ∘ f` or `f ; g` depending on orientation.
- **Parallel/tensor composition** = **horizontal juxtaposition**: two boxes placed side by side, with no connection between them.

```
   |   |          |         |   |
  [f] [g]   vs.  [f]   and [g] [h]
   |   |         [g]         |   |
                  |
```

### Linearized Text Notation

There is no universally adopted linearized ASCII form of string diagrams. The most common approach is to treat:
- Vertical composition as standard function composition (`;` or `∘`)
- Horizontal/tensor composition as `⊗` (see section 3)

Some computer algebra systems define string diagrams formally as well-typed formulae over a signature, where terms are generated by the signature, identity, composition, and tensor. But there is no standard "text string diagram" language — the whole point is they're graphical. Tools like `quipper` (Haskell), `Catlab.jl` (Julia), and `discopy` (Python) implement string diagrams computationally but use library calls rather than infix notation.

**Key insight:** In string diagrams, the *horizontal* axis is the tensor product (independent parallel processes), and the *vertical* axis is sequential composition. A connector that "spreads horizontally" in text notation would thus evoke the tensor product semantics.

## 3. Monoidal Category Product Operators

### Tensor Product ⊗ (U+2297, CIRCLED TIMES)

The primary monoidal operation. `A ⊗ B` means A and B exist simultaneously — resources are combined "in parallel." In a monoidal category, ⊗ is a functor `C × C → C` with a unit object I and associativity/unit isomorphisms.

```
A ⊗ B ⊗ C    -- three things held simultaneously
```

**When used:** Linear types (two resources used together), tensor networks, quantum circuits, systems of simultaneous constraints. The "multiplication" of a monoidal category. Does NOT imply one before the other.

**Visual:** Circled times. Heavier than desired (enclosed circle), but well-known.

### Direct Sum ⊕ (U+2295, CIRCLED PLUS)

Coproduct or biproduct in additive categories. `A ⊕ B` means "A or B" in some contexts (direct sum of vector spaces), or an exclusive OR in Boolean algebra, or the additive disjunction in linear logic. Semantically messier than ⊗.

```
A ⊕ B ⊕ C    -- additive combination, A+B+C as a coproduct
```

**When used:** Additive categories, linear logic (as the "plus" connective — choice controlled by the prover), XOR in digital logic. Typically means "one or the other" rather than "both simultaneously."

### Cartesian Product × (U+00D7)

Standard set-theoretic product. `A × B` means the set of all pairs (a,b). In category theory, the categorical product.

```
A × B × C
```

**When used:** When you want tuples of independent values, not tensor structure. Typically carries more "data" than ⊗ (which tracks resources).

### & (Ampersand — Linear Logic / Additive Conjunction)

In linear logic, `A & B` (pronounced "with" or "additive conjunction") means: you can have either A or B, but the *choice* is made by the environment. The context is shared — not split. Contrasts with ⊗ where the context is partitioned.

```
A & B & C    -- choose one, using a shared resource pool
```

**When used:** Linear logic, session types. Represents a service that can respond to either A or B (but not both simultaneously).

## 4. Profunctor and Span Composition

### Profunctors

A profunctor (also called a distributor or bimodule) `F : C ⇸ D` is a functor `D^op × C → Set`. The arrow `⇸` (U+21F8) distinguishes profunctors from ordinary functors.

Profunctor composition uses `∘` in nLab convention, but the underlying operation is a **coend**:

```
(G ∘ F)(e,c) := ∫^(d ∈ D) F(d,c) ⊗ G(e,d)
```

The coend "traces out" the intermediate variable d — this is exactly the same structure as relational composition: "there exists a middle term b such that (a,b) ∈ R and (b,c) ∈ S."

**For constraint systems:** Profunctors are more general than functions because they allow a many-to-many relationship between inputs and outputs. Their composition is intrinsically "relational" rather than functional.

### Spans

A span from A to B is an object X with maps `A ← X → B`. Span composition is done by pullback:

```
A ←— X —→ B ←— Y —→ C
           becomes
A ←— X ×_B Y —→ C
```

No special infix notation is standard for span composition; it inherits `∘` or `;` from its context.

### Z Notation Relational Composition Symbols

Z notation uses specific Unicode symbols for relational composition:

| Symbol | Unicode | Name | Meaning |
|--------|---------|------|---------|
| ⨾ | U+2A3E | Z Notation Relational Composition | R ⨾ S: apply R then S (diagrammatic order) |
| ⨟ | U+2A1F | Z Notation Schema Composition | Schema-level composition (fat semicolon) |

The fat semicolon ⨟ (U+2A1F) is visually heavier than `;` but makes the left-to-right reading explicit and is more visible than `·`.

## 5. Process Algebra Notations (CSP)

CSP (Communicating Sequential Processes, Hoare 1978, Roscoe) defines several composition operators:

| Operator | Symbol | Unicode | Meaning |
|----------|--------|---------|---------|
| Sequential | `;` | U+003B | P ; Q: P then Q (Q starts when P terminates) |
| Interleaving | `\|\|\|` | U+007C ×3 | P \|\|\| Q: fully independent, arbitrarily interleaved |
| Interface Parallel | `\|[X]\|` | U+007C, brackets | P \|[X]\| Q: synchronize on events in set X |
| Alphabetized Parallel | `\|\|` | U+007C ×2 | P \|\| Q: synchronize on shared alphabet |
| External Choice | □ | U+25A1 | P □ Q: environment picks first event |
| Internal Choice | ⊓ | U+2293 | P ⊓ Q: process picks nondeterministically |
| Hiding | `\` | | P \ X: hide events in X |

### Example Chains

```
-- Sequential: task pipeline
Authenticate ; Authorize ; Serve

-- Parallel with synchronization: all must agree on "commit"
Reader |[{commit}]| Writer |[{commit}]| Logger

-- Interleaving: independent agents
UserInput ||| NetworkIO ||| FileIO

-- Choice: process responds to one of two events
ReadRequest □ WriteRequest
```

**Key distinction:** `|||` (interleaving) is fully symmetric — P ||| Q = Q ||| P. `||` with shared alphabet is also symmetric. Sequential `;` is strictly ordered.

## 6. Linear Logic Connectives

Linear logic (Girard 1987) treats propositions as resources consumed exactly once. This eliminates contraction and weakening, giving a "resource-aware" logic.

| Symbol | Name | Type | Meaning |
|--------|------|------|---------|
| ⊗ | Tensor | Multiplicative conjunction | Both A and B, using separate resource pools (A and B consumed independently) |
| ⅋ | Par | Multiplicative disjunction | Dual of tensor; A or B from the same pool |
| & | With | Additive conjunction | Both A and B available, but same pool — choose one |
| ⊕ | Plus | Additive disjunction | Either A or B (prover's choice), same pool |
| ⊸ | Lollipop/Multimap | Linear implication | A consumed exactly once to produce B |
| ! | Of course / Bang | Exponential | A can be used freely (reintroduces weakening/contraction) |

### The Multiplicative/Additive Distinction

The critical distinction: **multiplicative** (⊗, ⅋) splits the context between subproofs; **additive** (& , ⊕) shares the context.

```
-- Tensor: I need a key AND a coin, from different pockets
key ⊗ coin ⊢ open_door

-- With: I can prove A or prove B, both proofs use the same resources
A & B    -- I have resources to do either, but not both at once

-- Chaining tensor: consume three resources in parallel
battery ⊗ motor ⊗ controller ⊢ robot_action
```

**For constraint systems:** ⊗ is the most relevant — it says "both constraints must hold, drawing on their own variable scopes." This is closer to a constraint conjunction than sequential piping.

## 7. Haskell / Functional Programming

### Function Composition `.` (period)

Standard Haskell function composition. Applicative (right-to-left) order.

```haskell
f . g . h     -- h runs first, then g, then f
-- type: (b -> c) -> (a -> b) -> (a -> c)
```

Visual: Very lightweight — just a dot — but tiny and easily confused with member access or decimal points.

### Arrow Composition `>>>` (triple chevron right)

From `Control.Arrow` and `Control.Category`. Left-to-right composition.

```haskell
f >>> g >>> h     -- f runs first, then g, then h
-- type: Arrow a => a b c -> a c d -> a b d
```

Visual: Three characters, strongly directional, reads in execution order. Used for composing `Arrow` instances including `Kleisli` arrows. In `Control.Category`, `(>>>)` works for any category.

### Kleisli Composition `>=>` (fish operator)

Composes monadic (Kleisli) arrows left-to-right.

```haskell
f >=> g >=> h     -- f runs first, threading monadic context
-- type: Monad m => (a -> m b) -> (b -> m c) -> (a -> m d)
```

Visual: Three characters, directional. The `>=>` is called the "fish operator" or "right fish." The left fish `<=<` reverses direction.

### Monoid Append `<>` (diamond / semigroup operator)

For any `Semigroup`, `<>` is the append/combine operation. Lists: concatenation. Maybe: first-Just. Functions: ???. Endomorphisms: composition.

```haskell
"hello" <> " " <> "world"    -- "hello world"
All True <> All False         -- All False
Sum 3 <> Sum 4                -- Sum 7
```

Visual: Two characters, symmetric-looking but slightly directional in Haskell (left-biased for Some types). Often used as a "combining" operator for monoid instances.

### Category Summary Table

| Operator | Direction | Associativity | Use |
|----------|-----------|---------------|-----|
| `.` | Right-to-left | Right | Pure function composition |
| `>>>` | Left-to-right | Left | Arrow/category composition |
| `>=>` | Left-to-right | Left | Kleisli (monadic function) composition |
| `<=<` | Right-to-left | Right | Reverse Kleisli |
| `<>` | Left-to-right | Left | Monoid/semigroup append |

## 8. Relational Composition

Relational composition: given R ⊆ A×B and S ⊆ B×C, compose to get T ⊆ A×C where (a,c) ∈ T iff ∃b. (a,b)∈R ∧ (b,c)∈S.

### Notation History

| Symbol | Unicode | Source | Notes |
|--------|---------|--------|-------|
| `;` | U+003B | Schröder 1895 | Standard in relational algebra, diagrammatic order |
| `∘` | U+2218 | Howie, algebraic semigroup theory | Applicative order — reverses text sequence |
| ⨾ | U+2A3E | Z Notation | "Relational composition," left-to-right |
| ⨟ | U+2A1F | Z Notation | "Schema composition" (fat semicolon) |
| ⨝ | U+2A1D | Database theory | JOIN — natural join of relations |
| ⋈ | U+22C8 | Database theory | BOWTIE — natural join variant |
| juxtaposition RS | — | Abstract algebra | No operator, treated as multiplication |
| `⊛` | U+229B | Some notations | Circled asterisk — occasionally used |

### Example Chains

```
-- Semicolon (Schröder, diagrammatic):
Parent ; Sibling ; Child    -- grandparent-uncle-cousin chain

-- Z notation relational composition:
isParentOf ⨾ isSiblingOf ⨾ isChildOf

-- Natural join:
Employees ⨝ Departments ⨝ Projects

-- Bowtie:
R ⋈ S ⋈ T
```

**Directional note:** The semicolon `;` and ⨾ are left-to-right (diagrammatic). The small circle ∘ is right-to-left (applicative). Juxtaposition varies by convention.

## 9. What Comes Closest to Symmetric, Left-to-Right, Lightweight, Horizontal Spread

Evaluating each approach:

| Notation | Symmetric? | L-to-R? | Lightweight? | Horizontal Spread? | Notes |
|----------|-----------|---------|--------------|-------------------|-------|
| ∘ | No | No (R-to-L) | Medium | No | Too directional |
| ; | No | Yes | High | No | Reads L-to-R but punctuation-feel |
| ⨟ (fat semicolon) | No | Yes | Medium | Slight | Better than `;` visually |
| ⊗ | Yes | Yes | Low (enclosed) | No | Circled symbol, heavy |
| & | Yes | Yes | High | Slight | Very common, may conflict |
| ⋈ (bowtie) | Yes | Yes | Medium | **Yes** | Symmetric, spread, joins two sides |
| ⨝ (join) | Yes | Yes | Medium | **Yes** | Explicit join semantics |
| ⟗ (full outer join) | Yes | Yes | Low | **Yes** | Very explicit but niche |
| \|\|\| | Yes | Yes | Low | **Yes** | Three chars, CSP interleaving |
| \|\| | Yes | Yes | Low | **Yes** | Two chars, parallel composition |
| <> | Near | Yes | High | Slight | Common monoid append |
| >>> | No | Yes | Medium | Yes | Directional arrow |

**Closest candidates for symmetric horizontal spread:**

1. **⋈ (U+22C8, BOWTIE)** — Visually symmetric, reads left-to-right, has horizontal extent, already means "natural join of two relational structures." A chain `A ⋈ B ⋈ C` reads naturally as "A joined with B joined with C." This is arguably the best existing symbol for constraint system composition.

2. **⨝ (U+2A1D, JOIN)** — The "official" Unicode join. Similar to bowtie but taller, used in database theory specifically for natural join. `A ⨝ B ⨝ C`.

3. **⟗ (U+27D7, FULL OUTER JOIN)** — Even more symmetric (the full outer join includes both sides). Very horizontal. But niche and unknown outside database contexts.

4. **\|\|\| (triple pipe)** — Three characters, but CSP users will recognize it as interleaving (symmetric, concurrent). `A ||| B ||| C` is established notation for independent parallel processes.

5. **~ ~ ~ or ~&~** — ASCII-only alternatives with horizontal spread. Not standard in any field but visually lightweight.

| Symbol | Unicode | Name | Visual Character |
|--------|---------|------|-----------------|
| ⋈ | U+22C8 | BOWTIE | Symmetric hourglass/butterfly, horizontal |

| ⁂ | U+2042 | ASTERISM | Three stars, decorative but wide |
| ⊕ | — | — | — |

### Top Candidates for "Compose These Constraint Systems Together"

**If you want maximum recognizability (from existing math):**
- `⋈` — Already means "join two relational structures." Has precedent. Symmetric. Horizontal.

**If you want tensor/resource semantics (both hold simultaneously):**
- `⊗` — Established in linear logic and monoidal category theory for simultaneous conjunction.

**If you want left-to-right pipeline feel:**
- `⨟` (fat semicolon) — Visually heavier than `;`, clearly diagrammatic order.
- `>>>` — Strong left-to-right convention from Haskell arrows.

**If you want a new symbol with the right visual shape:**
- `⟗` (full outer join, U+27D7) — Wide, symmetric, has the "join two sides" visual meaning.
- `⊛` (circled asterisk, U+229B) — Circled star, lighter than ⊗, less loaded semantically.
- `⟡` (white concave-sided diamond, U+27E1) — Open diamond shape, very lightweight, but no mathematical convention to fight against.

**If ASCII-only is required:**
- `<>` — Monoid append, widespread in Haskell/Elm/Rust. Visually symmetric.
- `|+|` or `|&|` — No standard meaning; the pipes give horizontal spread.
- `~` — Wave/tilde; used in some systems as "is similar to" or "approximately"; very lightweight.

---

## Summary: Design Recommendation for Evident

**The goal:** A symbol that means "compose these constraint systems together" — symmetric (not directional), reads left-to-right, visually lightweight, has horizontal spread.

**Best options ranked:**

1. **⋈ (bowtie, U+22C8)** — Best fit semantically and visually. Natural join of relational structures is essentially what constraint composition does (find tuples satisfying both). Symmetric, has horizontal spread (like two sides meeting in the middle), no directional bias. Chain reads naturally: `AddressConstraint ⋈ EmailConstraint ⋈ AgeConstraint`.

2. **⊗ (tensor, U+2297)** — Strong mathematical pedigree for "both simultaneously." Linear logic ⊗ is exact: both constraints must hold, drawing on their own variable scopes. Less visually spread but universally understood in type theory. Chain: `NonEmpty ⊗ Sorted ⊗ Unique`.

3. **⟗ (full outer join, U+27D7)** — Most visually spread of the join family. Looks like two triangles joined at a center. Unused in most programming contexts so no semantic baggage. Chain: `A ⟗ B ⟗ C`.

4. **`<>` (ASCII monoid append)** — If you want ASCII-only and Haskell/Rust familiarity. Already means "semigroup combine" which is similar to constraint conjunction. Chain: `constraint1 <> constraint2 <> constraint3`.

5. **`~` or `~~`** — If you want maximum visual lightness and have no use for established semantics. A tilde between terms is unobtrusive. But has no mathematical grounding.
