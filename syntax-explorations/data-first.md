# Syntax: Data-First (YAML)

## Philosophy

Programs are structured data first, executable logic second. A YAML document is a valid Evident program; an Evident program is a valid YAML document. This duality means programs can be parsed, diffed, merged, templated, and generated by ordinary data tools — `jq`, Ansible, Helm, Python dicts — without a dedicated parser. The tradeoff is deliberate: human ergonomics yield to machine composability. Variables are `?`-prefixed strings (Datalog convention). Claims are maps. Derivation trees are also maps. Everything is serializable, versionable, and introspectable at rest.

## 1. Factorial

```yaml
# factorial.evident.yaml
#
# Defines factorial as two named claims: the base case (0! = 1)
# and the recursive case (n! = n * (n-1)!).
# A query at the bottom asks for the value of 6!.

---
# --- RULES ---

- claim: factorial
  params: [?n, ?result]
  # Base case: factorial(0) = 1
  # No 'because' block means this is a self-evident axiom when params unify.
  alternatives:
    - because:
        - {op: "==", left: "?n", right: 0}
        - {op: "==", left: "?result", right: 1}

    - because:
        # Recursive case: n > 0, recurse on n-1
        - {op: ">",  left: "?n", right: 0}
        - {op: "-",  left: "?n", right: 1, bind: "?n_minus_1"}
        - claim: factorial
          params: ["?n_minus_1", "?sub_result"]
        - {op: "*",  left: "?n", right: "?sub_result", bind: "?result"}

---
# --- QUERY ---

- query: factorial
  params: [6, "?result"]
  # Ask: for what value of ?result is factorial(6, ?result) evident?
  # Expected answer: ?result = 720
```

**What this means:** The `alternatives:` key encodes disjunction — the runtime will try each `because:` block until one succeeds. The base case unifies `?n` with `0` and `?result` with `1`. The recursive case uses inline operator maps for arithmetic. Binding a computed value uses `bind:` as an output slot.

**Works well:** The base/recursive split is visually obvious; any YAML-aware tool can distinguish the two cases. The `alternatives:` vs `because:` symmetry cleanly maps to OR-of-ANDs normal form.

**Feels awkward:** Arithmetic steps like `{op: "-", left: "?n", right: 1, bind: "?n_minus_1"}` are verbose compared to an infix expression. A three-line computation in most languages becomes four separate map entries here.

## 2. Sorted List

```yaml
# sorted.evident.yaml
#
# A list is sorted if it is empty, a singleton, or its first element
# is <= its second element AND the tail is sorted.

---
# --- RULES ---

# A list is sorted if it has zero elements.
- claim: sorted
  params: ["?list"]
  alternatives:
    - because:
        - {op: "==", left: "?list", right: []}

    # A singleton list is always sorted.
    - because:
        - {op: "length", of: "?list", bind: "?len"}
        - {op: "==",    left: "?len", right: 1}

    # General case: head <= second element, and the tail is sorted.
    - because:
        - {op: "head",   of: "?list", bind: "?first"}
        - {op: "tail",   of: "?list", bind: "?rest"}
        - {op: "head",   of: "?rest", bind: "?second"}
        - {op: "<=",     left: "?first", right: "?second"}
        - claim: sorted
          params: ["?rest"]

---
# --- ASSERT (test data) ---

- assert:
    claim: my_list
    value: [1, 3, 5, 7, 9]

- assert:
    claim: bad_list
    value: [1, 5, 3, 7]

---
# --- QUERIES ---

- query: sorted
  params: ["?list"]
  where:
    - claim: my_list
      bind: "?list"
  # Expected: evident (1 <= 3 <= 5 <= 7 <= 9)

- query: sorted
  params: ["?list"]
  where:
    - claim: bad_list
      bind: "?list"
  # Expected: not evident (5 > 3 at position 1)
```

**What this means:** `assert:` introduces ground facts — a named claim whose value is already known. The `query:` block uses a `where:` sub-clause to bind `?list` from a named fact before passing it to the `sorted` claim. Structural list operations (`head`, `tail`, `length`) are expressed as operator maps.

**Works well:** `assert:` blocks read like a data fixture file, which domain experts can understand and edit without knowing anything about the inference rules. Separating data from rules is natural in YAML.

**Feels awkward:** The two-step `head`/`tail` dance to peek at the first two elements is cumbersome. A pattern-matching destructure (`[?first, ?second | ?rest]`) would be far more readable, but that requires syntax beyond plain YAML scalars.

## 3. HTTP Request Validation

```yaml
# http-validation.evident.yaml
#
# Validation rules for an HTTP API request.
# This file is designed so that a non-programmer (e.g. a security team member
# or API product owner) can edit the allowed methods, paths, and size limits
# without touching any code.
#
# The top-level 'config:' block is pure data. The 'claim:' blocks below
# reference it. A domain expert only needs to touch 'config:'.

---
# ============================================================
# CONFIGURATION — edit this section freely
# ============================================================

config:
  allowed_methods: [GET, POST, PUT, DELETE]
  max_body_bytes: 1048576          # 1 MB
  required_headers: [Content-Type, X-Request-ID]
  allowed_content_types:
    - application/json
    - application/x-www-form-urlencoded
  rate_limit:
    max_requests: 100
    window_seconds: 60

---
# ============================================================
# RULES — derived automatically from config above
# ============================================================

# A request is valid if ALL sub-claims are evident.
- claim: request_valid
  params: ["?request"]
  because:
    - claim: method_allowed
      params: ["?request"]
    - claim: body_size_ok
      params: ["?request"]
    - claim: headers_present
      params: ["?request"]
    - claim: content_type_allowed
      params: ["?request"]
    - claim: not_rate_limited
      params: ["?request"]

# Method must appear in the configured allowlist.
- claim: method_allowed
  params: ["?request"]
  because:
    - {op: "get",      of: "?request",        key: "method",  bind: "?method"}
    - {op: "contains", collection: {config: "allowed_methods"}, item: "?method"}

# Body must be within the configured size limit.
- claim: body_size_ok
  params: ["?request"]
  because:
    - {op: "get",  of: "?request",       key: "body_bytes", bind: "?size"}
    - {op: "<=",   left: "?size",         right: {config: "max_body_bytes"}}

# Every required header must be present.
- claim: headers_present
  params: ["?request"]
  because:
    - {op: "get",        of: "?request",  key: "headers", bind: "?headers"}
    - {op: "for_each",   in: {config: "required_headers"}, as: "?h",
       require:
         - {op: "contains", collection: "?headers", item: "?h"}}

# Content-Type must appear in the allowed list.
- claim: content_type_allowed
  params: ["?request"]
  because:
    - {op: "get",      of: "?request",    key: "headers",      bind: "?headers"}
    - {op: "get",      of: "?headers",    key: "Content-Type", bind: "?ct"}
    - {op: "contains", collection: {config: "allowed_content_types"}, item: "?ct"}

# Request count in the sliding window must be under the limit.
- claim: not_rate_limited
  params: ["?request"]
  because:
    - {op: "get",      of: "?request",            key: "client_id",   bind: "?cid"}
    - {op: "get",      of: {config: "rate_limit"}, key: "window_seconds", bind: "?window"}
    - {op: "get",      of: {config: "rate_limit"}, key: "max_requests",   bind: "?limit"}
    - {op: "count_requests_in_window",
       client: "?cid", window: "?window", bind: "?count"}
    - {op: "<", left: "?count", right: "?limit"}

---
# --- QUERY (used at runtime by the gateway) ---

- query: request_valid
  params: ["?request"]
  bind_from: incoming_request
  # If evident: allow the request through.
  # If not evident: the derivation tree identifies exactly which sub-claim failed.
```

**What this means:** The `config:` block at the top is deliberately separated from the inference rules. A security engineer can change `max_body_bytes` or add an entry to `allowed_methods` without touching any claim logic. Rules reference `{config: "key"}` as a first-class YAML value that the runtime resolves. The evidence tree on failure pinpoints the exact sub-claim (`method_allowed`, `body_size_ok`, etc.) that was not evident.

**Works well:** This is the strongest use case for YAML syntax. The config/rules split maps perfectly onto existing organizational workflows — the config block looks exactly like a Kubernetes ConfigMap or an Ansible vars file. Non-programmers can own the config layer with no training.

**Feels awkward:** `{op: "for_each", in: ..., as: "?h", require: [...]}` stretches the flat operator-map idiom; it's essentially a mini-loop expressed as a nested map, which doesn't have a clear YAML precedent and may confuse domain editors despite the rest of the file being approachable.

## 4. Graph Reachability

```yaml
# graph-reachability.evident.yaml
#
# Assert a directed graph as ground facts, then define reachability
# (direct edge or transitive path). Query which nodes are reachable from A.

---
# --- GRAPH DATA (asserted facts) ---
# Each edge is a self-evident claim with two params: [from, to].
# Adding or removing edges is a pure data edit — no rule changes needed.

- assert: {claim: edge, params: [A, B]}
- assert: {claim: edge, params: [A, C]}
- assert: {claim: edge, params: [B, D]}
- assert: {claim: edge, params: [C, D]}
- assert: {claim: edge, params: [D, E]}
- assert: {claim: edge, params: [F, G]}   # disconnected component

---
# --- RULES ---

# A node ?to is reachable from ?from if there is a direct edge.
- claim: reachable
  params: ["?from", "?to"]
  alternatives:
    # Base case: direct edge.
    - because:
        - claim: edge
          params: ["?from", "?to"]

    # Recursive case: there exists an intermediate node ?mid.
    - because:
        - claim: edge
          params: ["?from", "?mid"]
        - claim: reachable
          params: ["?mid", "?to"]

---
# --- QUERIES ---

# Which nodes are reachable from A?
- query: reachable
  params: [A, "?to"]
  collect: "?to"
  # Expected: [B, C, D, E]  (F and G are in a disconnected component)

# Is E reachable from A?
- query: reachable
  params: [A, E]
  # Expected: evident
  # Evidence tree would show: A->B->D->E  (or A->C->D->E, one valid path)

# Is G reachable from A?
- query: reachable
  params: [A, G]
  # Expected: not evident
```

**What this means:** Graph edges are pure `assert:` facts — compact single-line maps. The `reachable` claim has two `alternatives:` blocks encoding direct-edge and transitive-path cases. The `collect:` key on a query gathers all solutions for a variable into a list. The disconnected component `F->G` serves as a negative test.

**Works well:** The edge assertions are extremely clean — a list of single-line maps that a domain expert (e.g. a network engineer defining connectivity policy) could edit directly. The data/rules separation is sharp and the graph topology is immediately visible by scanning the asserts.

**Feels awkward:** The existential intermediate variable `?mid` in the recursive case has no explicit "there exists" keyword; it implicitly arises from unification, which may surprise readers unfamiliar with logic programming. A `exists: ["?mid"]` annotation on the `because:` block would clarify intent without adding syntax complexity.

## 5. FizzBuzz

```yaml
# fizzbuzz.evident.yaml
#
# The four FizzBuzz cases, modeled as four alternatives on a single claim.
# A query over the range 1..20 collects the output sequence.

---
# --- RULES ---

- claim: fizzbuzz
  params: ["?n", "?output"]
  # Four mutually exclusive cases, tried in order.
  # (Clause ordering is irrelevant to correctness in Evident, but
  #  the most specific case — divisible by both — should unify first
  #  for efficiency. The runtime may reorder; the 'priority:' hint advises it.)
  alternatives:

    - priority: 1
      # Divisible by both 3 and 5 → "FizzBuzz"
      because:
        - {op: "mod",  left: "?n", right: 15, bind: "?r"}
        - {op: "==",   left: "?r", right: 0}
        - {op: "==",   left: "?output", right: "FizzBuzz"}

    - priority: 2
      # Divisible by 3 only → "Fizz"
      because:
        - {op: "mod",  left: "?n", right: 3,  bind: "?r"}
        - {op: "==",   left: "?r", right: 0}
        - {op: "mod",  left: "?n", right: 5,  bind: "?r5"}
        - {op: "!=",   left: "?r5", right: 0}
        - {op: "==",   left: "?output", right: "Fizz"}

    - priority: 3
      # Divisible by 5 only → "Buzz"
      because:
        - {op: "mod",  left: "?n", right: 5,  bind: "?r"}
        - {op: "==",   left: "?r", right: 0}
        - {op: "mod",  left: "?n", right: 3,  bind: "?r3"}
        - {op: "!=",   left: "?r3", right: 0}
        - {op: "==",   left: "?output", right: "Buzz"}

    - priority: 4
      # Divisible by neither → the number itself as a string
      because:
        - {op: "mod",  left: "?n", right: 3,  bind: "?r3"}
        - {op: "!=",   left: "?r3", right: 0}
        - {op: "mod",  left: "?n", right: 5,  bind: "?r5"}
        - {op: "!=",   left: "?r5", right: 0}
        - {op: "to_string", of: "?n", bind: "?output"}

---
# --- QUERY ---

- query: fizzbuzz
  params: ["?n", "?output"]
  where:
    - {op: "range", from: 1, to: 20, bind: "?n"}
  collect:
    key: "?n"
    value: "?output"
  # Expected:
  #   1→"1", 2→"2", 3→"Fizz", 4→"4", 5→"Buzz",
  #   6→"Fizz", ..., 15→"FizzBuzz", ...
```

**What this means:** Each FizzBuzz case is one entry in `alternatives:`. The `priority:` hint advises the runtime to try more-specific cases first, even though correctness does not depend on ordering. The "divisible by neither" case must explicitly assert `?r3 != 0` and `?r5 != 0` because Evident has no negation-as-failure shorthand — every constraint is a positive claim about a computed value. The `collect:` on the query builds a key-value map of results.

**Works well:** The four cases are visually parallel and easy to audit — a domain expert can read "mod 15 == 0 → FizzBuzz" without knowing logic programming. Adding a new case (e.g. divisible by 7 → "Bazz") is a pure data append.

**Feels awkward:** The lack of a `not:` or `unless:` primitive forces the "divisible by neither" case to repeat two modulo computations that already appeared in earlier alternatives. This redundancy would compound badly in a larger case analysis (e.g. a 10-case FizzBuzz variant). A `guard:` key listing negative constraints would improve readability without breaking the data-first model.

## Overall Assessment

The YAML/data-first syntax excels in two specific scenarios: configuration-heavy domains (Program 3) and graph/fact-heavy domains (Program 4). When a program's primary content is *data* — edges, config values, fixtures — YAML's native strength makes the result genuinely readable and editable by non-programmers. The `assert:` / `claim:` / `query:` tripartite structure maps naturally onto existing infrastructure-as-code mental models.

The syntax struggles where computation is dense. Arithmetic (Programs 1 and 5) inflates dramatically: a single infix expression like `result = n * factorial(n-1)` expands to four operator maps. Structural matching on lists (Program 2) requires sequential `head`/`tail` operations that obscure what is conceptually a single pattern. The absence of a `not:` primitive creates awkward repetition in case-analysis problems.

The deepest tension is that YAML was designed for *configuration data with shallow nesting*; logic programs require *deeply nested, highly recursive* structure. Each recursive rule alternative adds another indentation level. At scale, a nontrivial Prolog program would produce YAML that is wide, deep, and hard to follow — precisely the opposite of what YAML is good at. This syntax is best suited as a *serialization target* for a friendlier surface syntax, or as a *domain-specific subset language* where the rule complexity is low and the data volume is high.
